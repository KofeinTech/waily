# WAIL-14 — Environment configuration

**Status:** Design
**Date:** 2026-04-27
**Author:** Claude (with kolyamriy)
**Jira:** https://improvs.atlassian.net/browse/WAIL-14

## What

A single `.env` file in the project root provides runtime configuration for the Flutter app. The app exposes those values through a typed, immutable config object reachable from anywhere via two top-level getters: `kEnv` (data) and `kEnvHelper` (sealed union with `isDev` / `isProd` flags). A dev-only banner overlay tells the developer when they are running against a dev environment.

## Why

The team needs environment-aware behavior (API URL, feature flags, dev indicator) without committing secrets and without complicating the project with per-flavor native build configs. A single `.env` plus `.env.example` is enough for the current stage; multi-environment installs on the same device are not a real need yet.

## Scope decision

The original Jira AC asked for three `.env` files, Android product flavors and iOS schemes. After scoping discussion (see WAIL-14 description "Scope decision"), we landed on the simpler model below. Native flavors / schemes can be added in a follow-up ticket if the need arises.

## Acceptance Criteria (revised in Jira on 2026-04-27)

1. `.env` lives in the project root, gitignored, loaded at app startup.
2. `.env.example` is committed as a template (documents all keys with placeholder values).
3. Environment-specific values (API URL, feature flags) are accessible throughout the app via a typed, immutable config object.
4. App displays an environment indicator (banner) when running in dev mode (`TYPE=DEV`).
5. Sensitive values are excluded from version control (`.env` in `.gitignore` — already covered).

## Architecture

### Folder layout

```
lib/core/env/
  env.dart              # main library file (imports + public API + sealed union)
  env_parser.dart       # part of 'env.dart' — extension _Parse on String
  env_variables.dart    # part of 'env.dart' — _EnvVariables, _EnvData
  env.freezed.dart      # generated by freezed
```

```
. (project root)
  .env                  # gitignored, developer creates from .env.example
  .env.example          # committed template
```

### Public API

```dart
Future<void> initEnv();   // idempotent; called once from main()
_EnvData get kEnv;        // typed values
_Env get kEnvHelper;      // sealed union with isDev / isProd extension
```

`_EnvData` is internal (underscored) but exposed via the `kEnv` getter — callers receive a typed object, not a `Map<String, String>`. Equality is provided through `Equatable`.

### Sealed union

`_Env` is a freezed sealed union with two variants — `_Env.dev(_EnvData)` and `_Env.prod(_EnvData)` — selected at construction time by the `TYPE` value inside `.env`. A static `_instance` cache keeps the union singleton-like (re-reading `kEnvHelper` returns the same instance).

```dart
@freezed
sealed class _Env with _$Env {
  static _Env? _instance;

  factory _Env() {
    assert(_parser.isInitialized, 'kEnv not initialized, call initEnv() first');
    return _instance ??= switch (_EnvVariables.envType.toUpperCase()) {
      'PROD' => _Env.prod(_env),
      _      => _Env.dev(_env),
    };
  }

  const factory _Env.dev(_EnvData data) = _Dev;
  const factory _Env.prod(_EnvData data) = _Prod;
}

extension EnvX on _Env {
  bool get isDev => this is _Dev;
  bool get isProd => this is _Prod;
}
```

### Parser extension

```dart
final _parser = dotenv;

extension _Parse on String {
  String get env => _parser.env[this] ?? '';
}
```

Allows clean call sites: `'API_BASE_URL'.env`. Missing keys return `''` rather than throw, mirroring the reference pattern from another team project.

### Typed variables

```dart
abstract class _EnvVariables {
  static String get envType      => 'TYPE'.env;            // DEV | PROD
  static String get apiBaseUrl   => 'API_BASE_URL'.env;
  static bool   get enableLogging => 'ENABLE_LOGGING'.env.toLowerCase() == 'true';
}

_EnvData get _env => _EnvData._(
  apiBaseUrl: _EnvVariables.apiBaseUrl,
  enableLogging: _EnvVariables.enableLogging,
);

class _EnvData extends Equatable {
  const _EnvData._({required this.apiBaseUrl, required this.enableLogging});
  final String apiBaseUrl;
  final bool enableLogging;
  @override
  List<Object?> get props => [apiBaseUrl, enableLogging];
}
```

`TYPE` itself is intentionally **not** part of `_EnvData`. Type-of-environment is exposed as a sealed union (`kEnvHelper.isDev`), not as a string field — that makes wrong comparisons (`kEnv.envType == 'dev'`) impossible.

### Boot sequence

```dart
// lib/main.dart
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await initEnv();
  runApp(const WailyApp());
}
```

```dart
// lib/app.dart (new — currently main.dart holds the app widget directly)
class WailyApp extends StatelessWidget {
  const WailyApp({super.key});
  @override
  Widget build(BuildContext context) {
    return MaterialApp(
      debugShowCheckedModeBanner: false,
      darkTheme: darkTheme,
      themeMode: ThemeMode.dark,
      home: const Scaffold(),
      builder: (context, child) =>
          DevEnvBanner(child: child ?? const SizedBox.shrink()),
    );
  }
}
```

### Dev banner

```dart
// lib/core/ui_kit/components/dev_env_banner.dart
class DevEnvBanner extends StatelessWidget {
  const DevEnvBanner({super.key, required this.child});
  final Widget child;

  @override
  Widget build(BuildContext context) {
    if (!kEnvHelper.isDev) return child;
    return Banner(
      message: 'DEV',
      location: BannerLocation.topEnd,
      child: child,
    );
  }
}
```

Uses Flutter's built-in `Banner` — diagonal corner stripe, no custom theming needed. In prod the wrapper short-circuits to `child` so the widget tree shape remains the same.

## Data flow

`main()` → `initEnv()` (loads `.env` into `dotenv`) → `runApp(WailyApp())` → `MaterialApp.builder` wraps every screen with `DevEnvBanner`. Application code reads `kEnv.apiBaseUrl` (or other typed getters) at the point of use; `_Env` is constructed lazily on first access and cached.

When future tickets add Dio/Retrofit (WAIL-? — out of scope here), the API client will receive `kEnv.apiBaseUrl` at construction time. Today the value is unused at runtime, but the typed accessor exists.

## Error handling

| Failure | Behavior |
|---|---|
| `.env` missing in bundle | `flutter_dotenv` throws on `load()`; `runApp` never runs; developer sees a clear stack trace. |
| `.env` malformed | Same as above. |
| `kEnv` accessed before `initEnv()` | `assert` in `_Env()` factory fails in debug; in release the parser is empty so all keys return `''` (degraded but not crashing). |
| Missing key inside `.env` | `'KEY'.env` returns `''`. Type-coerced (`bool`) values get a safe default (`false` for unknown). Acceptable because `.env.example` documents the required key set. |
| Repeated `initEnv()` calls | Idempotent — early return when already initialized. |

## Testing strategy

`flutter_dotenv` exposes `dotenv.testLoad(fileInput: '...')` for unit tests, allowing us to feed env content inline without touching the file system.

**`test/core/env/env_test.dart`:**
- `TYPE=DEV` → `kEnvHelper.isDev == true`; `kEnv.apiBaseUrl` matches input.
- `TYPE=PROD` → `kEnvHelper.isProd == true`.
- Unknown `TYPE` value → defaults to `dev`.
- `ENABLE_LOGGING=true` → `kEnv.enableLogging == true`; `false` / missing → `false`.
- Reset the singleton (`_Env._instance = null`) in `setUp` of each test; expose a private `@visibleForTesting` reset hook in `env.dart`.

**`test/core/ui_kit/components/dev_env_banner_test.dart`:**
- With `TYPE=DEV` loaded → tree contains `Banner` with `message: 'DEV'`.
- With `TYPE=PROD` loaded → tree does NOT contain `Banner`; child renders unchanged.

Tests use `dotenv.testLoad` in `setUp` so they never depend on a real `.env` file. Coverage exists for the two AC-relevant behaviors (typed access, dev-only indicator).

## Risks

1. **Singleton cache (`_Env._instance`) makes tests order-dependent.** Mitigated by a `@visibleForTesting` reset hook reset in `setUp`.
2. **Library-private API (`_EnvData`, `_Env`) cannot be referenced from outside `env.dart`.** Intentional — callers are forced to use `kEnv` / `kEnvHelper`. Public type names (`EnvData`, `Env`) would be cleaner; we keep them private to match the team's reference pattern. Re-evaluate if future code needs to type-annotate against `_EnvData` directly.
3. **`flutter_dotenv` reads from the asset bundle.** Changes to `.env` require a hot restart (not hot reload). Document this in README.
4. **Freezed 3.x and a non-redirecting `factory _Env()` next to redirecting `factory _Env.dev/prod`.** The reference snippet from the team uses freezed 2.x, where mixed factories work. If freezed 3.x rejects this, fall back to a top-level constructor function: `_Env _buildEnv()` containing the switch / cache logic, called by both `kEnvHelper` and tests. The sealed union itself stays the same. Verify during implementation.
